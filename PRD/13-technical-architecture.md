# 13 · Technical Architecture & mock→backend strategy

How the demo is built so it **feels real now** and **becomes real later** without a rewrite.
Guiding constraint (Abhran): *modular, easy to build, little custom development.*

---

## 1. Stack

| Layer | Choice | Why |
|-------|--------|-----|
| Framework | **Next.js (App Router) + TypeScript** | Routing, RSC, easy deploy; backend-ready. |
| UI | **Tailwind + shadcn/ui** | Pre-built accessible primitives; low custom code. |
| Charts | **Recharts** | Covers gauges/timeseries/bars; declarative. |
| Tables | **TanStack Table** | Sorting/filtering/pagination for call logs & queues. |
| State/data | **TanStack Query** over a typed **data-access layer** | Caching, refetch, loading/error states. |
| Backend (later) | **Supabase** (Postgres + Auth + RLS + Storage) | Mirrors call DB; RLS matches role model. |
| Audio | HTML5 audio + waveform lib | Recordings in Call Detail. |

---

## 2. The data-access layer (the key to "mock now, backend later")

Every screen reads through **typed adapters**, never directly from a data source.

```
UI (components) → hooks (useCalls, useCostRollup, useLiveOps, …)
                → DataSource interface (typed methods + Zod schemas)
                → adapter:  MockAdapter   |   SupabaseAdapter
```

- One interface per domain: `CallsSource`, `CostSource`, `MetricsSource`, `InfraSource`,
  `LiveSource`, `FallbackSource`, `FlagSource`, `BusinessSource`.
- Selected by env: `DATA_SOURCE=mock | supabase`.
- **Same Zod-validated shapes** for both adapters → swapping is config, not refactor.
- Pure domain logic (cost computation, margin, MRR) lives in `/lib/engine` and is shared by the
  mock seed generator and the future ETL.

---

## 3. Mock data strategy

- **Deterministic seed** (fixed RNG seed) → reproducible demo; "regenerate" script for variety.
- `/lib/seed` builds 3 orgs, ~6 projects, agents, and N calls with realistic usage → runs through
  `/lib/engine` to produce `call_cost` + `period_rollup` (true cost/revenue/margin).
- Infra series (k8s/ELB) generated by series-spec generators keyed to template vars (doc 06).
- **LiveSource** mock = client ticker (doc 07); interface ready for SSE/WebSocket later.
- Seed can be emitted as JSON (mock) **or** SQL/Supabase seed (when backend is on) from the same code.

## 4. Module structure (modular, low-custom)

```
/app
  /(dashboard)
    /cost  /performance  /calls/[callId]  /issues
    /live  /infra/kubernetes  /infra/elb
    /controls/fallbacks  /controls/thresholds  /controls/flags
    /business  /qa-bench
/components        # shared: KpiStrip, panels (Gauge/Stat/Timeseries/BarGauge/Log), DataTable, chips
/lib
  /engine          # cost/revenue/margin/MRR — pure functions
  /data            # DataSource interfaces + mock/supabase adapters
  /seed            # deterministic generators
  /auth            # can(user, action, resource) policy
  /design          # tokens (doc 14)
```

- Reusable panel components make infra pages **declarative arrays of specs** (doc 06 §3).
- A single `<ModuleScope>` wrapper applies role + org/project filtering uniformly.

---

## 5. AuthZ

- `app_user` carries `role` (`superadmin`\|`pm`\|`dev`\|`financial`) + `grants[]` (`{scopeType:
  "org"|"project", scopeId}[]`) — SuperAdmin carries none (implicit `*`). See doc 01, doc 20.
- Category predicates in `lib/auth/policy.ts` (`canSeeFinancials`, `canSeeOpsModules`,
  `isSuperAdmin`) plus scope helpers in `lib/auth/scope.ts` (`effectiveOrgIds`,
  `effectiveProjectIds`) are the single source of truth, used by UI (hide/disable/filter) and future
  RLS (enforce).
- Demo: SuperAdmin provisions `PM`/`Dev`/`Financial` identities via Access Management (doc 20), then
  previews any of them through the **View-as** switcher (doc 01 §2.1) — no real login yet.
- Backend: Supabase RLS policies mirror the same role + effective-scope logic by `org_id`/`project_id`;
  real auth (SSO/magic-link) replaces "View-as" preview with each person logging in as themselves
  (doc 20 §6, open question).

---

## 6. Real-time

- Only `/live` updates (15–30s). Demo via `setInterval` ticker behind `LiveSource`.
- Backend path: Supabase Realtime / SSE polling of `chat_conversations` — same interface.

---

## 7. Non-functional

- **Performance:** rollup tables for charts (no scanning raw calls client-side).
- **States:** every data surface has loading / empty / error.
- **Theming:** light (default, per Platform palette) + dark; tokens in `/lib/design` (doc 14).
- **A11y:** shadcn/Radix primitives; keyboard nav on tables/menus.
- **i18n-ready:** UI strings centralized (English only for v1, per requirement).

---

## 8. Deploy
- Vercel (Next.js) for the demo; Supabase project when backend is enabled. Env-flag switch between
  mock and live. No secrets in the mock path.
